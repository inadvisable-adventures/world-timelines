# PostgreSQL + PostGIS schema for world timelines data

## Summary

Design the `db/` directory: SQL files that fully define a PostgreSQL schema
capable of holding everything currently served by the app — entries,
historical eras, lanesets, and lanes — faithful to the existing TypeScript
types and TSV/JSON formats, with UUID primary keys and `last_updated`
timestamps on entries, lanesets, and lanes. This plan defines the schema only;
seeding/loading from the existing files is `db-init-script.md`.

## Source-of-truth data model (from the current codebase)

- `web-client/src/types/index.ts` — canonical types: `HistoricalEvent`,
  `EventLocation` (5-way union: point/polygon/multipolygon/path/circle),
  `EventDate`, `Laneset`, `Lane`.
- `ingester/src/types.ts` — near-duplicate of the above (ingestion side);
  schema should match both since they describe the same on-disk format.
- On-disk formats actually read by the app today:
  - `web-client/public/data/collected_entries.sample.tsv` — 43 entries, 17
    tab-separated columns: `id title locations start_year start_month
    start_day end_year end_month end_day start_expr end_expr calendar
    uncertainty_years category infobox_type description tags`. `locations`
    and `tags` are inline JSON.
  - `web-client/public/data/historical_eras.tsv` — 114 rows, **identical
    17-column shape** to entries (confirmed by inspecting the file directly).
    The client (`app-root.ts:267-293`, `parseErasTsv`) reduces each row to
    `{id, title, startYear, endYear, source}`, deriving `source` from the
    first tag ending in `-history` (default `'world-history'`). Critically,
    `end_year`/`end_month`/`end_day`/`end_expr` are **always populated** in
    the raw file — the TS `endDate: EventDate | null` is a derived notion
    computed by the parser (`tsv-parser.ts:82-89`: null only when
    `endYear === startYear`), not something encoded in storage. The schema
    therefore stores end fields as `NOT NULL`, matching the file, not the
    derived in-memory type.
  - `web-client/public/data/lanesets.json` — `{ lanesets: Laneset[] }`,
    generated by `web-client/scripts/gen-lanesets.mjs`. Each `Laneset` is
    `{id, name, description, lanes: Lane[]}`; each `Lane` is `{id, name,
    description, geometry: MultiPolygon, bbox: [minLng,minLat,maxLng,maxLat],
    eraSources?: string[]}`.

## Scoping decision: what goes in Postgres

**In scope** (this is "world timelines" content, per the TODO item):
entries (`collected_entries.sample.tsv`), eras (`historical_eras.tsv` — same
row shape as entries), lanesets + lanes (`lanesets.json`).

**Out of scope**: `infobox-catalog.tsv`, `infobox-by-category/*.tsv`, and
`ingester/ingest.config.tsv`. These are ingestion-pipeline tooling data (a
scan of which Wikipedia infobox templates map to which category, and ingest
filter config) — not timeline content. `world-110m.geojson` (vendored Natural
Earth source data, an input to `gen-lanesets.mjs`, not app-served content)
also stays out.

**Key modeling decision — one `entries` table, not a separate `eras` table**:
since `historical_eras.tsv` is byte-for-byte the same 17-column shape as
`collected_entries.sample.tsv`, and an "era" is really just an entry with
`category = 'historical_period'` and a tag ending in `-history`, both are
imported into a single `entries` table. This is the faithful representation
of the source data (not a new invented distinction) and means one importer
code path handles both files. The app-level "is this an era" distinction
(driving whether the timeline renders it as a band vs. a marker) stays a
query-time filter (`category = 'historical_period' AND tags && ARRAY[...
'-history' tags]`), exactly mirroring what `parseErasTsv` already does
client-side.

## Schema

`gen_random_uuid()` is built into PostgreSQL core since v13 — no `pgcrypto`
extension needed for UUIDs. PostGIS is needed for geometry columns.

```sql
CREATE EXTENSION IF NOT EXISTS postgis;

CREATE TABLE lanesets (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug         TEXT NOT NULL UNIQUE,       -- e.g. 'continents' (Laneset.id today)
  name         TEXT NOT NULL,
  description  TEXT NOT NULL,
  last_updated TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE lanes (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  laneset_id   UUID NOT NULL REFERENCES lanesets(id) ON DELETE CASCADE,
  slug         TEXT NOT NULL,              -- e.g. 'africa' (Lane.id today)
  name         TEXT NOT NULL,
  description  TEXT NOT NULL,
  geometry     geometry(MultiPolygon, 4326) NOT NULL,
  bbox_min_lng DOUBLE PRECISION NOT NULL,
  bbox_min_lat DOUBLE PRECISION NOT NULL,
  bbox_max_lng DOUBLE PRECISION NOT NULL,
  bbox_max_lat DOUBLE PRECISION NOT NULL,
  era_sources  TEXT[] NOT NULL DEFAULT '{}',
  last_updated TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (laneset_id, slug)
);
CREATE INDEX lanes_geometry_gix ON lanes USING GIST (geometry);

CREATE TABLE entries (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug              TEXT NOT NULL UNIQUE,   -- content-derived id today, e.g. 'great-pyramid-giza'
  title             TEXT NOT NULL,
  start_year        INTEGER NOT NULL,
  start_month       SMALLINT NOT NULL,
  start_day         SMALLINT NOT NULL,
  end_year          INTEGER NOT NULL,
  end_month         SMALLINT NOT NULL,
  end_day           SMALLINT NOT NULL,
  start_expr        TEXT NOT NULL,
  end_expr          TEXT NOT NULL,
  calendar          TEXT NOT NULL DEFAULT 'gregorian'
                      CHECK (calendar IN ('gregorian','julian','islamic','hebrew','unknown')),
  uncertainty_years DOUBLE PRECISION NOT NULL DEFAULT 0,
  category          TEXT NOT NULL
                      CHECK (category IN ('person','event','place','artifact',
                        'pol_mil_organization','business','historical_period',
                        'concepts','other')),
  infobox_type      TEXT NOT NULL DEFAULT '',
  description       TEXT NOT NULL DEFAULT '',
  tags              TEXT[] NOT NULL DEFAULT '{}',
  last_updated      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX entries_start_end_year_idx ON entries (start_year, end_year);
CREATE INDEX entries_category_idx ON entries (category);
CREATE INDEX entries_tags_gin ON entries USING GIN (tags);

CREATE TABLE entry_locations (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entry_id     UUID NOT NULL REFERENCES entries(id) ON DELETE CASCADE,
  ordinal      SMALLINT NOT NULL,           -- position in the original locations[] array (0 = primary anchor)
  kind         TEXT NOT NULL CHECK (kind IN ('point','polygon','multipolygon','path','circle')),
  geometry     geometry(Geometry, 4326),    -- Point for 'point'/'circle' (circle center), Polygon, or MultiPolygon; NULL for 'path'
  radius_km    DOUBLE PRECISION,            -- only for kind='circle'
  waypoints    JSONB,                       -- only for kind='path': [{lat,lng,t?}, ...], preserves optional per-point t
  uncertain    BOOLEAN NOT NULL DEFAULT false,
  label        TEXT,
  UNIQUE (entry_id, ordinal)
);
CREATE INDEX entry_locations_geometry_gix ON entry_locations USING GIST (geometry);
CREATE INDEX entry_locations_entry_id_idx ON entry_locations (entry_id);
```

### Mapping notes

- `entries.slug` / `lanes.slug` / `lanesets.slug` preserve today's
  content-derived string ids (needed so the seed script and any future
  re-import can upsert idempotently by natural key, and so URLs/DSL that
  reference a laneset/lane by name keep working). `id` (UUID) is the new
  surrogate primary key requested by the TODO item.
- `entry_locations.geometry` uses the generic PostGIS `geometry` type (not
  fixed to one subtype) because a table mixes Point/Polygon/MultiPolygon rows
  by `kind`; the app/importer is responsible for writing the right subtype.
- `path` locations don't map onto a single PostGIS geometry cleanly because
  each waypoint may carry an optional fractional-progress `t` — forcing this
  into a `LineStringM` would require a value for every point and lose the
  "optional" nature. Storing `waypoints` as JSONB preserves exact fidelity
  from the TS type and is simple; `path` locations are not populated by any
  current data anyway.
- `circle` locations store their center as a PostGIS `Point` in `geometry`
  and `radius_km` alongside it, rather than materializing a buffered polygon
  — this preserves the exact semantics (center + radius) instead of an
  approximation, and a buffered polygon can always be derived at query time
  with `ST_Buffer` if ever needed.
- No `entry_locations.last_updated` — the TODO item asks for UUID +
  last-updated on entries, lanesets, and lanes specifically; a location's
  freshness is implied by its parent entry's `last_updated`, and this is a
  child table, not one of the three named entity types.

## Affected files (new)

- `db/schema.sql` — the DDL above, in dependency order (extension → lanesets
  → lanes → entries → entry_locations), idempotent via `CREATE TABLE IF NOT
  EXISTS` / `CREATE EXTENSION IF NOT EXISTS`.
- `db/README.md` — what's in this directory, the one-time `brew install
  postgresql@16 postgis` prerequisite (from `install-local-postgres.md`), and
  a pointer to the init script (`db-init-script.md`) that applies this schema
  and seeds it.
